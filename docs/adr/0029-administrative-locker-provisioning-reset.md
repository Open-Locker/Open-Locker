# ADR-0029: Administrative locker provisioning reset

## Status

Proposed

## Date

2026-07-13

## Context

Locker provisioning can fail or leave a device in a state that operators cannot
recover through the Filament admin panel. Issue #110 and PR #120 add an
administrative action that rotates the provisioning token and allows the locker
client to provision again after its local provisioning state and credentials
have been cleared manually.

The existing implementation stores the provisioning token as plaintext on the
`locker_banks` row. Initial tokens are generated by an Eloquent `creating` hook;
locker-bank creation and token generation are not event-sourced. A clean event
replay therefore cannot reconstruct existing provisioning tokens.

The first PR draft put the newly generated token into
`LockerProvisioningReset`. That would retain every rotated token in plaintext in
the append-only event store and its backups. It would also make only reset tokens
replayable while initial tokens remain outside event sourcing.

The current MQTT device username is the locker-bank UUID. Resetting can delete
the corresponding `mqtt_users` row, and the Mosquitto go-auth cache is disabled,
so later authentication and ACL checks for the missing user are denied. Deleting
the row does not guarantee that Mosquitto immediately closes an established TCP
session. Recreating the same UUID username during re-provisioning may allow an
old session to pass ACL checks again without authenticating with the new
password.

This ADR deliberately limits #110/#120 to recovery from failed provisioning. It
does not define complete revocation of a compromised device identity.

## Decision

### Recovery semantics

Treat the Filament action as an **administrative provisioning recovery**, not as
a complete security revocation workflow.

A dedicated backend service owns the reset. The Filament resource delegates to
that service and does not call the aggregate directly. The service:

1. Requires an authenticated actor with `lockerbank.configure`.
2. Serializes concurrent resets for the locker bank and performs all local
   database changes in one transaction.
3. Generates a new 64-character provisioning token and writes it directly to the
   `locker_banks` row.
4. Deletes the current MQTT user for the locker-bank UUID.
5. Records `LockerProvisioningReset` with only:
   - locker-bank UUID;
   - actor user ID;
   - reset timestamp.
6. Exposes a generic translated failure notification in Filament while logging
   technical exception details server-side.

The event is audit metadata. It does not contain the token and does not make
provisioning-token state replayable. It is included in the admin audit log with
the actor, locker bank, and timestamp.

### Read-model state reset

The transaction updates the locker bank to:

- set `provisioned_at` to `null`;
- set `connection_status` to `offline`;
- set `connection_status_changed_at` to the reset timestamp;
- clear `last_heartbeat_at`;
- clear `last_config_sent_at`;
- clear `last_config_sent_hash`;
- clear `last_config_ack_at`;
- clear `last_config_ack_hash`.

Compartment contents, history, configuration, and last-known door state remain
unchanged. Because the bank is offline, retained door state is not evidence of a
current live connection.

After re-provisioning, configuration remains dirty until a fresh `apply_config`
command is sent and acknowledged.

### Secret handling

For #120, keep the existing plaintext token storage and admin copy workflow on
the `locker_banks` row. Do not add HMAC storage or encryption in this PR.

Provisioning tokens and registration topics containing them must not be written
to backend or locker-client logs. The HMAC/storage redesign is deferred to #159.

### MQTT session boundary

Deleting the MQTT user is required, and tests must prove that the auth and ACL
endpoints deny the deleted identity. This ADR does not require a custom
Mosquitto plugin, broker kick, or application-level blanket rejection of inbound
messages while `provisioned_at` is `null`.

An application-level unprovisioned guard would cover only the temporary reset
window and would not solve reuse of the UUID username after re-provisioning.
Unique per-provisioning MQTT identities, migration of existing credentials,
durable denial of old sessions, retained-message handling across credential
generations, and optional connection eviction are deferred to #161.

Operators must clear the locker client's local provisioning state and saved MQTT
credentials, install the new provisioning token, and restart the client.

## Alternatives Considered

### Store the plaintext token in the reset event

- Pros: The projector can restore rotated tokens during replay.
- Cons: Every token remains in the append-only event store and backups; initial
  tokens still cannot be replayed.
- Why not chosen: Partial replayability does not justify durable plaintext
  secret retention.

### Migrate to HMAC token storage in this PR

- Pros: The backend can validate tokens without retaining recoverable token
  material.
- Cons: Changes token lookup, admin delivery, migration of unprovisioned banks,
  key management, and operational procedures.
- Why not chosen: It is a separate security and workflow decision tracked in
  #159.

### Encrypt tokens in the event store

- Pros: Preserves token recovery and replay with the application key.
- Cons: Retains decryptable long-lived secrets and introduces key rotation,
  backup, and replay complexity.
- Why not chosen: The system does not currently guarantee token replay, so the
  additional secret-management complexity is not justified.

### Add broker kick or unique MQTT identities in this PR

- Pros: Can prevent an old connected device from regaining access after
  re-provisioning.
- Cons: A broker kick requires maintained broker integration; unique identities
  change the backend/client MQTT contract and require legacy credential
  migration.
- Why not chosen: #120 is limited to administrative recovery. The durable MQTT
  identity design is tracked in #161.

### Reject all inbound messages while unprovisioned

- Pros: Adds defense in depth during the temporary reset window.
- Cons: Does not distinguish old and new sessions after the UUID username is
  recreated and does not solve retained or delayed messages across generations.
- Why not chosen: It adds incomplete policy to this PR without establishing a
  durable identity boundary.

## Consequences

### Positive

- Administrators can recover a locker from failed provisioning.
- Rotated tokens are not copied into the event store.
- Reset actions have an actor-aware audit trail.
- Unauthorized callers cannot bypass Filament visibility to trigger a reset.
- The UI immediately reflects that the locker is offline and its previous
  configuration acknowledgement is stale.
- Existing provisioning and MQTT contracts remain compatible.

### Negative

- Provisioning tokens remain plaintext in the locker-bank table.
- Token state remains intentionally non-replayable.
- Reset requires manual changes and restart on the locker client.
- An established MQTT TCP session is not guaranteed to be disconnected.
- Reusing the UUID username during re-provisioning does not provide durable
  revocation of a potentially compromised old session.

### Risks

- Concurrent reset or registration requests could issue conflicting tokens or
  credentials. Mitigation: serialize resets with a database lock and cover
  repeated/concurrent behavior with tests.
- Failure between token rotation, MQTT-user deletion, and event persistence
  could leave partial state. Mitigation: keep local database changes and event
  persistence in one transaction and test rollback behavior.
- Token-bearing topics may leak through shared logging. Mitigation: redact
  registration topics in backend and locker-client logging and add log
  assertions.
- Operators may interpret reset as secure device revocation. Mitigation: label
  the action and documentation as administrative recovery and reference #161
  for the durable credential lifecycle.

## Rollout / Migration

1. Add the reset service, server-side permission check, transactional state
   cleanup, MQTT-user deletion, and actor-aware audit event.
2. Remove token-bearing event properties and redact token/topic logging.
3. Add tests for authorization, rollback, audit data, token absence, MQTT auth
   and ACL denial, state cleanup, repeated reset, and actual re-provisioning.
4. Keep the existing locker-client provisioning workflow and MQTT username
   format for compatibility.
5. Resolve #159 and #161 independently before claiming at-rest token hardening
   or durable MQTT credential revocation.
